# Applied Energistics: Wireless Nexus 实现计划

> 状态：设计已确认，功能实现进行中。
>
> 原则：按阶段实现；每一阶段完成并验证后再进入下一阶段。发现 AE2、GT5U 或 ModularUI2
> 的实际 API 与本文假设不一致时，先更新本文并说明原因，不静默改变设计。

## 1. 目标范围

本项目计划实现三部分功能：

1. **ME 无线网络控制器**
   - 继承并保留原版 AE2 `TileController` 的控制器行为。
   - 控制器群遵守 AE2 原有结构限制，包括最大 `7 x 7 x 7` 以及同一个 AE
     网络不能存在多个独立控制器群。
   - 控制器群可设置一个网络显示名称；网络的存档内唯一身份由放置时生成的 UUID 保证。
   - 扫描控制器群可用于无线连接的频道容量。

2. **AE 无线连接器**
   - 玩家可以从有权限的已命名无线网络中选择一个进行连接。
   - 权限由目标 AE 网络的安全终端，即 `ISecurityGrid`，在服务端判断。
   - 每个已成功连接并获得容量的连接器最多向本地 AE 网络提供 32 个频道。

3. **GregTech 仓室与 AE 设备集成**
   - 仅对同时满足 `MTEHatch` 和 AE 相关接口的 MTE 提供无线网络能力。
   - 有 ModularUI2 UI 时，在 GT UI 内添加按钮。
   - 没有自身 UI 时，玩家交互后直接打开公共网络选择界面。

## 2. 已确认的设计决策

### 2.1 控制器基础

- 无线控制器 Tile 继承 AE2 `TileController`，不重新实现原版控制器的结构校验、
  频道传播和网络状态逻辑。
- 同一个 AE 控制器 Grid 中只允许一个无线控制器；出现多个时沿用 AE 的
  `CONTROLLER_CONFLICT` 状态，使整组控制器显示冲突并停止频道分配。
- 实现前先确认 AE2 控制器代码是否存在对原版 Block 实例的精确比较；如果有，使用
  最小范围 Mixin 让自定义控制器参与原版判定，不复制整套控制器实现。
- 网络名称属于整个控制器群/AE 网络，不属于某一个控制器方块。

### 2.2 网络身份与名称

- 每个控制器在放置时生成稳定的内部 `UUID`；控制器群形成网络后沿用该网络身份，显示名称只是可修改属性。
- 控制器放置时写入一个默认显示名称。默认名称可以使用本地化名称加 UUID 短片段，避免玩家首次打开时看到空名称。
- 连接器和 MTE 保存网络 UUID，不直接以名称作为绑定键，因此控制器改名不会导致连接丢失。
- 显示名称允许所有可见字符，包括中文、空格和标点；只拒绝控制字符与换行。
- 为避免 ModularUI2 列表被撑破，初步限制为 32 个 Unicode 字符；UI 宽度不足时截断显示，并在 tooltip 中显示完整名称。
- 显示名称不是网络身份，也不作为持久化绑定键。列表排序使用显示名称后再使用 UUID，允许同名网络仍可被区分。
- 名称修改、可见字符/长度校验和绑定操作全部由服务端处理。

### 2.3 无线频道容量扫描

- 扫描单位是控制器群中每个控制器方块的六个面，包括无线控制器、普通控制器以及
  `TileCreativeEnergyController` 自充能控制器。
- 一个合格的面对无线容量贡献 32 频道。
- 以下面不贡献无线容量：
  - 面外相邻方块是同群或其他控制器；
  - 面外相邻方块是其他 AE 设备；
  - 该面存在 AE 连接并承载或消耗频道。
- 总无线容量为 `合格面数量 * 32`。
- AE 设备识别优先使用 AE 的 Grid/Host/Part API，不维护容易遗漏的方块白名单。
- 放置、破坏、邻居变化、控制器群变化和 AE Grid 状态变化会使扫描结果失效；服务端合并同一
  tick 内的重复通知后重算，避免大型控制器群频繁完整扫描。
- 客户端只接收名称和容量等展示数据，不自行计算权威结果。

### 2.4 容量分配

- 一个无线连接器最多承载 32 频道，按 AE 无线连接上实际经过的频道数动态申请容量，不预扣 32。
- GT 直连仓室不申请固定的 32 频道，而是按其原本 AE Grid 节点的频道需求申请容量。
- 所有无线端点按连接界面保存的优先级参与容量分配；优先级高的端点先分配，优先级相同时使用
  网络 UUID、维度和坐标等稳定键排序，避免每 tick 抖动。
- 当容量下降或多个区块同时加载导致超额时，无法获得完整请求容量的端点进入明确的“容量不足”状态，
  不静默绕过限制。
- 连接器断开、移除、目标网络离线或区块卸载时释放租约。
- 连接器本地侧通过 AE Grid 节点提供最多 32 频道，不绕过 AE 原生频道计算。
- GT 仓室连接成功后，其 ME 连接面仍然是一个普通 AE 连接面；从该面接出的线缆按 AE 原生规则
  产生和传递频道，普通机器/普通线缆默认最多 8 个频道，不因为无线连接而变成 32 频道。

### 2.5 AE 安全权限

- 网络列表只展示当前玩家可访问的网络。
- 玩家提交绑定请求时再次通过目标网络 `ISecurityGrid` 校验，不能信任客户端传来的网络 UUID。
- 默认以 AE 的 `BUILD` 权限作为“允许把设备接入网络”的权限；实现阶段会根据当前 AE2
  版本源码确认准确枚举和无安全终端时的原版语义。
- 绑定记录保存发起绑定的玩家身份。安全设置变化、目标网络重新上线或绑定恢复时需要重新校验；
  无法证明仍有权限时应断开，而不是继续使用旧授权。

### 2.6 GT UI 按钮位置

- **不使用**截图外侧 Cover 按钮的 `Column`，也不修改 `GTBaseGuiBuilder#createCoverTabs()`。
- 按钮放入 GT 面板内部右下角的 `createBottomRightCornerFlow`。
- 该 Flow 当前承载黄色 GT Logo、编程电路槽以及部分机器的额外槽位/按钮；无线按钮预期排列在
  这些控件左侧，实际顺序需要在游戏中确认。
- 通用注入点是
  `MTETieredMachineBlockBaseGui#createBottomRightCornerFlow(ModularPanel, PanelSyncManager)`
  的返回值。
- `MTEBasicMachineBaseGui` 和 `MTEDigitalTankBaseGui` 调用 `super`，原则上由通用注入覆盖。
- `MTEHatchInputBusMEGui` 覆盖该方法且不调用 `super`，需要单独注入。
- `MTEBetterJukeboxGui`、`MTESolarGeneratorGui` 等禁用右下角 Flow 的特殊 UI，不强行恢复默认
  Flow；只有目标 MTE 符合接入条件时才设计局部兼容入口。
- 不以 `MTEHatch#buildUI` 作为通用 Mixin 目标，因为基类没有这个方法，具体 MTE 的 UI
  构建方式也并不统一。

## 3. 计划中的模块划分

类名可以在实现时按现有代码风格调整，但职责边界保持如下：

```text
cn.dancingsnow.ae_wireless_nexus
├── block/                 无线控制器、无线连接器方块
├── tile/                  对应 TileEntity 与 AE Grid 节点生命周期
├── network/
│   ├── identity/          网络 UUID、名称校验和控制器群身份
│   ├── persistence/       WorldSavedData 与 NBT 数据结构
│   ├── capacity/          控制器面扫描、容量与租约分配
│   └── service/           查询、绑定、安全检查和生命周期协调
├── gui/                   ModularUI2 公共选择界面与控制器命名界面
├── integration/gregtech/  MTE 资格判断、适配器与 UI 打开逻辑
├── mixin/gregtech/         GT UI 和无 UI 交互注入
├── registry/              Block、TileEntity、Item 与配方注册
└── client/                图标、渲染和仅客户端 UI 资源
```

### 3.1 持久化模型

使用存档级 `WorldSavedData` 保存无线网络索引和稳定身份，数据至少包括：

- 网络 UUID；
- 显示名称与用于排序/搜索的派生键（如实现需要）；
- 控制器群所在维度和规范锚点坐标；
- 最近一次确认在线的状态信息；
- 必要的数据版本号，供以后迁移。

连接器和 MTE 自身 NBT 保存：

- 目标网络 UUID；
- 发起绑定的玩家身份；
- 当前绑定配置，不保存瞬时 `IGrid`、`IGridNode` 或 TileEntity 引用。

控制器群加载后向服务注册，卸载只标记离线；确认控制器被破坏或群身份变化后才删除/迁移记录，
避免普通区块卸载误删名称。

## 5. 实现前必须确认的问题

以下问题不影响保存总体计划，但进入对应阶段前必须明确：

1. **MTE 的 AE 接口覆盖范围**
   - 资格条件已经确定为 `MTEHatch` 与 AE 相关接口同时满足。
   - 具体纳入判断的接口列表仍需按当前 GT5U/AE2 源码确认，避免把仅有相似方法名的对象误判为 AE 端点。

2. **GT 直连端点的频道占用**
   - 连接器最多承载 32 频道，但按 AE 无线连接的实际频道用量申请容量。
   - GT MTE 直连按原生 AE Grid 节点频道需求申请容量；其 ME 连接面和外接线缆遵循普通 AE
     连接规则，普通机器默认最多 8 频道。

3. **名称规则**
   - 内部唯一 ID 已确定为放置时生成并持久化的 UUID。
   - 显示名允许所有可见字符（含中文），初步限制 32 个 Unicode 字符；需要在实际 UI 中确认
     是否应进一步缩短或仅在展示层截断。

4. **容量不足时的优先级**
   - 连接界面提供优先级选项，并将其和网络 UUID 一起持久化。
   - 优先级相同的端点按稳定、可复现的键分配；具体优先级上下限需要结合 ModularUI2
     数字控件和网络同步字段确认。

## 6. 变更纪律

- 本文件获确认后才开始功能实现。
- 每次只完成一个阶段或一个可独立验证的子阶段。
- 源码查找优先使用 IDEA MCP；IDEA 查找失败后再检查 Gradle cache 中的 sources JAR。
- 模组类 Mixin 使用 GTNH late mixin；early mixin 只用于原版类，并要求项目同时具备 coremod 入口。
- 不复制 AE2/GT5U 的大段实现；优先继承、调用公开 API，必要时使用最小范围 Mixin。
- 不把客户端 UI 状态当作权限或容量的事实来源。
- 不修改无关文件，不覆盖工作区中用户已有的变更。
